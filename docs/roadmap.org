#+title: UA4K Roadmap

This is a lightweight working roadmap, not a release promise.

* Immediate

- Playtest =Crash Landing= and =Dockstep= on laptop and phone.
- Tighten confusing level descriptions and level ordering.
- Keep the public site focused on =games/polished/= and the clearest toys.
- Add one or two hand-written notes to each polished game explaining its hook.
- Keep mobile controls usable, but optimize first for keyboard play.
- Cancel stale GitHub Pages deploys when rapid pushes supersede them.
- Treat the =SUP= level in =game.txt= as a useful quality bar: it has a
  compact board, a legible trick, and enough mirror/pushing structure that
  brute force is not a cheap substitute for understanding the route.

* Near Term

- Add curated screenshots or richer board previews to the site index.
- Improve =docs/creating-games.md= with a full tiny tutorial game.
- Add more parser diagnostics for common authoring mistakes.
- Decide whether =games/toys/= should all remain public by default.
- Replace the prompt-based level jump with a real level picker, and avoid
  accidental keypresses skipping the next level's description.
- Add shareable solution replays; the solver already emits compact key
  strings, so the smallest useful version is "copy replay / paste to watch".
- Audit older games for rules that can now be simplified with =REPEAT=.

* Later

- Add a browser workflow for editing a board and sharing it as a URL fragment.
- Consider a small hosted puzzle portal only after the static site and format
  docs are stable.
- Revisit game-level metadata, since =TITLE= and =BY= currently attach only
  after a level exists.
- Consider a better data format for site builds than generated JavaScript.
- Bring the Emacs frontend closer to browser parity: tick games need a timer
  loop, and =_init= rules should not be silently ignored.
- Consider small DSL primitives only when a concrete game asks for them:
  negative pattern tests (=absent=), declared character classes (=CLASS=), or
  capture wildcards for Sokoban-style goal tiles.

* Not Now

- Do not rewrite the parser in Rust unless Python parser performance or
  packaging becomes a real blocker.
- Do not publish the candidate games under =games/experiments/= until they
  have been played by a human. Exact solver proofs and mutation audits are
  necessary authoring checks, not substitutes for playtesting.
- Do not spend time simplifying =games/toys/life.txt= under the old "24
  neighbor rules" premise. Current =life.txt= already uses =FOR= + =ROTATE= +
  =REPEAT= for neighbor counting; any further cleanup needs a specific,
  observed authoring pain.
- Do not build a full web rule editor before the game format is easier to
  explain and validate.
- Do not add an in-game hint button or player-facing solver UI. The solver
  stays an offline authoring and testing tool (=solve_level.js=,
  =verify_minmoves.js=); puzzles are meant to be struggled with.
- Do not rename game files away from =.txt=. A dedicated extension would buy
  Emacs =auto-mode-alist= convenience but would obscure that games are plain
  text anyone can edit. =games/.dir-locals.el= gives Emacs the mode without
  the rename; revisit only if the format ever needs to be recognized outside
  this repo.
