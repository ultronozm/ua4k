#+title: UA4K Cleanup Ledger

This ledger records the current cleanup direction before the project is shared
more widely.

* Pass 2 (2026-07)

Findings from a whole-repo review. Ordered roughly by execution.

** DONE Hygiene
- Delete the disabled digit-wildcard branches and dead =digitMap= work in
  =ua4k.js= (original purpose is forgotten; the dead map was still computed on
  every rule application).
- Remove =emacs/game.el= (dead 2024 prototype, unrelated to the current
  frontend).
- =scratch/modular-schmodular.*= stays: it does not belong to the game, but it
  is deliberately kept until it has a better home.
- Make =make-data.py= back up an unparseable =gamesData.js= before starting
  from ={}= instead of silently discarding all previously compiled games.
- Derive the golden-snapshot default set from =tests/snapshots/*.json= so
  snapshots can never silently fall out of coverage again (five were orphaned:
  =drone-gate-*=, =lantern-trail=, =signal-relay=). Add a =clockwise= snapshot.

** DONE Solver as test harness
- Extract the fake-DOM/VM bootstrap shared by =tui_play.js=,
  =solve_level.js=, and =scratch_smoke.js= into =node_harness.js=.
- Add =replay_level.js=: step a key sequence against a level and print each
  board state. This was the essential authoring instrument for Clockwise.
- Add =verify_minmoves.js=: BFS-check every =MINMOVES= claim in every game,
  with state caps and skip-with-warning for games using =RANDOM= rules. Wire
  into =make check=. This doubles as the runtime smoke suite the previous pass
  asked for: every claimed level is exercised through the real runtime.

** DONE CI runs the checks
=pages.yml= only built and deployed the site; =make check= ran nowhere in CI.
Add a check job and make deploy depend on it.

** DONE Compiler internals
- =golden_snapshots.py= and =parser_diagnostics.py= now import
  =compile_game= like every other consumer instead of copying =make-data.py=
  to a tempdir and shelling out.
- Compile each game once per =build-all-assets.py= run instead of three times.
- Collapse the four identical ATOMIC/MATCH1/TRY_ALL/RANDOM directive handlers.

** DONE Format: game-level metadata and per-level goals
- =TITLE=/=DESCRIPTION=/=BY= before the first board now attach to the game.
  The site prefers them over its hand-maintained summary table.
- =GOAL=/=VOID= after a board attach to that level and replace the global
  patterns for that level (mirroring how =TICK= overrides). Implemented in
  both engines (=ua4k.js= and =ua4k.el=). This removes the hidden-status-row
  workaround Clockwise used to satisfy game-global goals vacuously.

** DONE Site and authoring docs
- Board preview (first board, colored) per game on the site index.
- =docs/level-design.md=: the solver-driven authoring method.
- =games/.dir-locals.el= so game files open in =ua4k-dsl-mode= in Emacs
  without renaming away from =.txt= (see roadmap note on extensions).

** Deferred from this pass
- Replace the =let gamesData = ...;= text contract with JSON plus a JS shim.
- Shareable solution replays and URL-fragment scratch boards.
- Solver-derived difficulty annotations on the site.
- =Clockwise= stays WIP despite its verified pars; mechanical verification is
  not the same thing as puzzle quality.
- =games/toys/life.txt= no longer has the old "24 neighbor rules" problem:
  neighbor counting is already expressed with =FOR=, =ROTATE=, and =REPEAT=.
- The =SUP= level in =games/polished/game.txt= is a good regression example
  for "interesting by thought, not just by BFS": full BFS over the general
  runtime can explode, while the intended route is to clear row 3, move the
  slash mirror to the far right, and fire north into the goal.

* Pass 1

** Remove Rust parser material

The Rust parser was only a stub and created a false impression that there were
two parser implementations to understand. The Python compiler remains the
source of truth.

** Fix the Dockstep rename fallout

=drone-swarm.txt= was renamed to =dockstep.txt=. Docs, test defaults, Emacs
defaults, and snapshots should use Dockstep.

** Put game files in clear buckets

- =games/polished/=: public-facing candidates.
- =games/toys/=: working toy/demo games.
- =games/wip/=: playable but not curated mechanic sketches.
- =games/experiments/=: jokes or odd experiments that are not in the normal
  curated set.
- =games/legacy/=: old format experiments that the current compiler is not
  expected to handle.

Initial classification:

- polished: =game=, =crash-landing=, =dockstep=
- toys: =bet=, =snake=, =tetris=, =turing=, plus smaller working demos
- WIP: =ice-slides=, =lantern-trail=, =signal-relay=, =phase-shift=,
  =drone-gate-*=
- experiments: =pong=
- legacy: =rpg=

** Keep local browser launch simple

The project should continue to support:

- compile with Python
- open an HTML file directly
- optionally ship a standalone browser artifact without requiring a web server

The accumulative =gamesData.js= workflow can stay for local development. The
cleaner share path should be generated static assets from =build-web.py=.

** Add a Makefile

There should be one obvious local quality command. It should run parser
diagnostics, golden snapshots, a build check, and a small runtime/solver smoke
test.

** Add a static public site

The first online version should be a GitHub Pages-compatible static site:

- one index page
- one page per published game
- generated data files checked at build time but not committed
- no arbitrary browser-side puzzle submission yet

This gives people a way to try the games before the project takes on portal or
editor complexity.

** Improve authoring docs

Keep =docs/dsl-spec-current.md= as the exact current behavior, but add a more
approachable guide for writing small games.

* Follow-Up Work

** Fix game-level metadata

Today =TITLE=, =BY=, and =DESCRIPTION= attach only after a level exists. That
makes top-level titles in files like =ice-slides.txt= effectively ignored. The
format should probably grow explicit game-level metadata, or the parser should
document and enforce the current level-only behavior more clearly.

** Harden browser rendering

The browser runtime used to render some game-authored data via =innerHTML=.
Board and metadata rendering should stay on text nodes or equivalent escaping,
especially before accepting any browser-authored puzzle input.

** Add scratch levels

For existing games, allow a player to paste or edit only the board grid and run
it against the checked-in rules for that game. This is a smaller, safer feature
than a full browser rules editor:

- no custom rules
- no generated JavaScript
- validate board size, shape, and character set
- save scratch boards locally in the browser
- keep full game editing as a later project

** Add runtime smoke tests

Golden snapshots protect compiler output, not runtime behavior. Add a small
browser or Node runtime smoke suite with one scripted move sequence per curated
game.

** Design hosted puzzle sharing later

A hosted puzzle portal needs more than static files:

- untrusted puzzle validation
- resource limits
- safe rendering
- share URLs
- possibly a browser-side editor and diagnostics
- a versioned compiled data contract
